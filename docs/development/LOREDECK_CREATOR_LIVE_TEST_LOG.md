# Deck Maker Live Test Log

Concise running log for real SillyTavern Deck Maker tests against `http://127.0.0.1:8000/` with `SAGA_ALLOW_PROVIDER_CALLS=1`.

## 2026-06-14

- Built out the `live-creator` smoke path in `tools/scripts/smoke-live-st-cdp.mjs` to drive Deck Maker intake through Scope Brief, Story Outline, title batches, Context/Tag planning, Lorecard drafting, Pending Review, Pack Health, finalization, Library/file verification, and cleanup.
- Real One Piece Arlong Park run reached Scope Brief but exposed fragile DOM-text waits after provider completion. Changed key waits to use Deck Maker state probe instead of overlay text.
- Repeated One Piece runs exposed stale active smoke jobs and late provider responses rehydrating into new sessions. Added cancellation/reset checks before starting a new live run and made reset validation ignore `Pending Review 0`.
- Real runs exposed slow post-approval UI rerenders and duplicate Deck Maker overlay surfaces. Hardened the click helper with bounded polling and all-matching-overlay lookup.
- Latest live One Piece run exercised Scope Brief, Story Outline, all title batches, one approved title, Context/Tag planning, one Lorecard draft, Send to Review, Accept All, and Pack Health. It bound at finalization because `Finalize Anyway` opens a confirmation dialog that the smoke runner did not confirm. Patched the runner to confirm coverage acknowledgement and wait for `Finalize as Custom Loredeck` to become enabled before finalization.
- Longer live One Piece run used seven real provider units through all title batches, then bound after approving the first title because another first-overlay text wait missed the visible `Plan Context and Tags` action. Replaced that wait with Deck Maker state/button-label probe.
- Next One Piece run hit a transient provider failure on Story Outline: two attempts ended with `No message generated` while the UI exposed `Retry Failed`. Added retry-aware generation waits that click `Retry Failed` once for Scope Brief, Story Outline, title batches, Context/Tag planning, and Lorecard drafting.
- Full One Piece run reached Pack Health and confirmed `Finalize Anyway`, but finalization stayed blocked on `Acknowledge coverage`. Root cause: `normalizeLoredeckCreatorJob()` dropped `coverageFinalizeAcknowledgement`, so the coverage acknowledgement vanished on project normalization. Added the field to the persisted Deck Maker job model and a regression assertion.
- Post-fix One Piece run produced another full title pass, including a retry on one title batch, then hit a redundant timing gate before `Plan Context and Tags`. Removed that gate and now rely on the polling click helper for the next action.
- A short rerun showed the Scope Brief approval state can flip before the `Draft Story Outline` button is actually mounted. Changed brief and outline approval waits to require the next actionable button before continuing.
- Full installed-extension sync was required before the live run could test the current workspace source. SillyTavern was serving an older installed Saga checkout, so mixed source files created false runtime failures until `src`, `styles`, and `manifest.json` were overlaid into the installed extension.
- The latest full One Piece run completed the real-provider path through Scope Brief, Story Outline, two title batches, title approval, Context/Tag planning, one Lorecard draft, Send to Review, Accept All, Pack Health, coverage acknowledgement, and finalization. File-level verification found a valid finalized Custom deck with one entry, no pending changes, Custom-pack entry sources, retired Deck Maker entry markers, finalization provenance, and the Generated source removed from the Library.
- The same run exposed a post-finalization UI bind: after file-level finalization succeeded, the live page could remain on `Finalizing...` and CDP evaluation timed out before the metadata editor/library UI settled. The smoke runner now records that as a distinct finalization UI-settle finding instead of letting it masquerade as only a cleanup failure.
- Cleanup exposed a second live-profile bind. The UI delete path could not run while the page was stuck, and Node-side files API cleanup received `403` without the browser session cookie. The runner now has a scoped filesystem fallback controlled by `SAGA_ST_USER_FILES_DIR`/`SAGA_ST_DATA_DIR`, and it removes linked Deck Maker project storage rows as well as finalized/generated Loredeck payload rows.
- Added the product-side finalization guard: once the Custom deck is durably created, post-save validation and Generated draft retirement are timeboxed, the busy button is restored before UI refresh/open-editor work, and the Deck Maker workbench overlay is closed before opening the finalized Custom metadata editor.
- Interrupted `live-creator-onepiece-fsfallback` run confirmed the filesystem cleanup fallback can remove the finalized Custom payload and indexed Deck Maker project payload after the page is wedged. It also exposed a fallback gap: `saga-creator-index.v1.json` still held the smoke project pointer after deleting the project file. The fallback now scrubs linked Creator index projects, active/last job pointers, and project files; the live profile was cleaned afterward.
